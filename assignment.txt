The Three Approaches to AI-Assisted Development 
Each approach below is a complete methodology for turning an app idea into a working product using AI tools. 
They differ in what artefact you create before asking AI to write code. In the assignment, you will use all three 
on the same problem and compare results. 
Approach 1: PRD-Driven Development 
Best for: Complex apps with multiple stakeholders, when you need the AI to understand the full product vision 
upfront. 
Process 
1. Write a comprehensive PRD before touching any AI tool: problem statement, target users, features 
(MoSCoW prioritisation), user flows, technical constraints, non-functional requirements 
2. Feed the entire PRD as system context to Claude/ChatGPT in a single conversation 
3. Ask AI to generate the complete codebase in one pass, referencing the PRD for decisions 
4. Iterate by updating the PRD and re-prompting for specific sections

Strengths: Complete context = fewer clarification rounds. AI makes consistent decisions. Good for handoff to other 
developers. 
Watch out for: Large PRDs can exceed context windows. AI may hallucinate features not in the PRD. Token cost is 
high per prompt. 
Approach 2: Architecture-First Development 
Best for: Technical apps with complex data flows, when you need clean separation of concerns and modular 
code. 
Process 
1. Create architecture diagrams before prompting: component tree, state management plan, API schema, data 
models, folder/file structure 
2. Use Excalidraw or draw.io to create visual diagrams – export as images 
3. Feed architecture documents as context, then ask AI to implement one module at a time 
4. Each prompt references the architecture diagram: “Implement the UserService module as defined in the 
architecture – it handles authentication and profile management” 
5. Stitch modules together with integration prompts 
Strengths: Modular, clean code. Easy to debug one module at a time. Lower token cost per prompt (smaller 
context). Parallel development possible. 
Watch out for: Requires upfront technical design skill. Integration between modules can be tricky. AI may deviate 
from the architecture if not reminded. 
Approach 3: User-Story-Driven Development (Iterative) 
Best for: MVPs and prototypes where requirements may evolve. When you want incremental, testable 
progress. 
Process 
1. Write user stories with acceptance criteria: “As a user, I want to log my daily steps so I can track my fitness 
over time. Acceptance: form validates input 0–100,000; saves to local storage; shows in history list.” 
2. Prioritise stories into a backlog (most critical first) 
3. Implement one story at a time with AI. Each prompt includes: the story, acceptance criteria, and a reference 
to the existing codebase 
4. Test each story before moving to the next. Fix issues within the same conversation 
5. Add stories incrementally – requirements can evolve as you see the product take shape 
Strengths: Testable at every step. Requirements can evolve. Works well with Replit Agent’s checkpoint model. 
Lowest risk of wasted work. 
Watch out for: Can lead to spaghetti code if architecture is not considered. Later stories may require refactoring 
earlier code. Context window fills up over many iterations. 
Assignment 4: Three-Approach Comparison Study 
Work Period: Mon, Apr 7 – Wed, Apr 9 (3 working days) 
Submission Deadline: Wednesday, April 9 
Objective 
Take a single, moderately complex app specification and build it three times – once with each approach above. 
Compare the results across quality, speed, cost, and maintainability. This is a structured learning exercise, not a 
race. 
App Specification (use for all three approaches) 
Avantro Labs  •  Hubballi, Karnataka  •  www.avantrolabs.com 
“Daily Habits Tracker” – a web app where users can define custom habits (e.g., “Drink 8 glasses of water”, 
“Walk 10,000 steps”), mark them complete each day, view streak data, and see a weekly/monthly summary 
with simple charts. Include user authentication (email/password) and cloud data storage. 
What to Submit for Each Approach 
• Artefact created: The PRD, architecture diagrams, or user stories document 
• Prompts used: Complete conversation log or prompt sequence (copy-paste from Claude/ChatGPT) 
• Output: The generated code (can be partial – focus on quality over completeness) 
• Evaluation: Rate each approach on: code quality (1–5), time spent (hours), number of prompts/iterations, 
estimated token usage, ease of debugging 
Final Comparison Report 
Write a 1–2 page report summarising: which approach worked best for this app and why, which approach you 
would use for a simple prototype vs a production app, and your recommended default approach for Avantro’s 
AI-native development workflow. 
Deliverables 
1. Three artefact documents (PRD, architecture diagrams, user stories) 
2. Three prompt/conversation logs 
3. Three code outputs (GitHub repos or folders) 
4. Comparison report (1–2 pages) 
5. Live demo of the best version to reporting manager
